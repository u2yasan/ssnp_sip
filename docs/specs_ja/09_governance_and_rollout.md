# 09. ガバナンスと段階導入

## 位置づけ
SSNP は、protocol rewrite ではなく、network-stability support program として導入すべきである。

## 導入フェーズ
### Phase 1
監視、登録、公開ランキング、通知のみ。

### Phase 2
外部プールからの手動または半手動の報酬配布。

### Phase 3
ガバナンス承認を前提とした自動計算・自動配布。

## ガバナンス原則
将来、protocol-level economic integration に進む場合は、
別個のガバナンス議論として扱わなければならない。

## 運営責任
ポータルは、責任主体が明確な組織のもとで運営されなければならない。
その主体は以下のいずれかとする。

- 専用の法人
- 正式な運営委員会
- 権限が定義された明示的な運営主体

その責任主体は最低でも以下に責任を持つ。
- 規約とポリシー変更
- 資金管理
- 異議申立てと紛争処理
- 最終的な分類判定
- 緊急介入

## AI と自動化の境界
AI エージェントと自動化は以下を扱ってよい。
- 監視
- 異常検知
- 通知送信
- ランキング計算
- レポート生成と Issue 起票

AI エージェントと自動化に最終権限を持たせてはならない領域は以下である。
- 資金支払い承認
- Same Operator Group の最終認定
- オペレーター失格確定
- 例外承認
- 異議申立ての裁定

## 低運用負荷の設計目標
SSNP が目指すべきなのは、無責任な無人化ではなく、低運用負荷化である。
資金を入れるだけで責任問題が消えるわけではない。
現実的な目標は以下である。

- 日常運用は大半を自動化
- センシティブな判断は人間承認
- 重要判断はすべて記録し、レビュー可能にする

## コミュニケーションルール
SSNP は常に以下として説明する。
- harvesting reduction ではない
- protocol change ではない
- CPU-performance contest ではない
- popularity reward system ではない

## MVP の立場
最初のリリースは、報酬原資が承認されなくても価値を持つべきである。
監視・登録レイヤーが報酬なしでは無意味なら、設計が間違っている。
