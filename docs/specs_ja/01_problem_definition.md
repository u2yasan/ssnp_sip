# 01. 問題定義

## 主問題
Symbol は安定した finalization のために有効な voting node の継続参加へ依存しているが、
その健全性を可視化し、対処可能にする共通運用枠組みが不足している。

## 現在の運用ギャップ
1. Voting node の継続性は重要だが、継続運用に焦点を当てたインセンティブは弱い。
2. Voting key は更新漏れで失効し、実効的な voting 参加が静かに失われうる。
3. ノード健全性の共通公開基準が存在しない。
4. 運用支配はドメイン、事業者、委託先の背後で集中しうる。
5. 監視や通知の運用はオペレーターごとにばらついている。

## NEM 型の枠組みをそのまま持ち込めない理由
旧 NEM Supernode Program は、computing power、ping、bandwidth のような
controller 主導の性能試験を重視していた。

しかし Symbol で本当に重要なのは「どれだけ速いマシンか」ではない。
重要なのは「このノードが今、実際に voting を継続できる状態にあるか」である。

## Symbol で重視すべきもの
- current epoch に対して有効な voting key
- チェーン追従の継続性
- finalization 追従の継続性
- endpoint 到達性
- 継続的な運用健全性

## リスク定義
これは主として報酬設計の問題ではない。
これは **ネットワーク運用耐性** の問題であり、将来の報酬配分に接続しうる基盤設計の問題である。
