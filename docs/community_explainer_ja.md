# SSNP コミュニティ向け説明資料（日本語）

[English](community_explainer.md) | 日本語

注記: この文書は日本語版です。対外的な共有や英語話者を含む議論では英語版も併読すること。

## 一言で言うと

SSNP は、**高品質な Symbol Voting Node を支えるためのネットワーク安定化プログラム**です。

## なぜ必要か

今の Symbol では、finalization は Voting Node の継続運用に依存している一方で、次の仕組みが不足しています。

- 運用品質の公開評価
- Voting Key 更新忘れの予防
- 単一オペレーターや運用代行業者への集中抑制
- 障害や期限切れの早期通知

## SSNP がやること

SSNP は以下を導入します。

- ノード登録
- 独立監視
- 性能ベースランキング
- 寡占防止フィルタ
- 運用通知
- 別枠原資による上位 Qualified Node への報酬

## SSNP がやらないこと

SSNP は以下を行いません。

- 既存ハーベスト報酬の削減
- ブロック報酬の変更
- コンセンサスルールの変更
- 人気や委任集中への報酬付け

## なぜ全 Voting Node に均等配分しないのか

均等配分は一見フェアですが、実際には悪いインセンティブを生みます。

- 低品質ノードにも報酬が出る
- 改善圧力が弱い
- 分散が進まない

SSNP は、**Qualified な性能** と **分散性** を評価して報います。

## なぜ寡占防止が必要か

寡占防止ルールが弱いと、1人のオペレーターや1つの運用代行業者が複数枠を取れてしまいます。見かけ上は分散していても、実際には同じ障害ドメノード群になります。

そのため、ドラフトでは

- Same Operator Group あたり 1 枠
- 同一 registrable domain あたり 1 枠

を基本としています。

## なぜ通知機能が重要か

大きな障害の多くは予防可能です。

- Voting Key 失効
- TLS 証明書期限切れ
- ノード停止の放置
- 同期遅延

SSNP は、落としてから罰するのではなく、**落ちる前に知らせる** ことも重視します。

## 初期導入

最初は以下だけで十分です。

1. 監視
2. ランキング
3. 通知

その後に、別枠原資からの報酬へ進みます。

## 一番重要なメッセージ

SSNP は、単なる報酬制度ではありません。

**Symbol の運用脆弱性を減らすための仕組み**です。
