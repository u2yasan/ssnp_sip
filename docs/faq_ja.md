# SSNP FAQ: 反対論点10個とカウンター

[English](faq.md) | 日本語

このFAQは、SSNP に対して想定される主要な反対論点を整理し、
それぞれに対する実務的なカウンターを示すものである。

前提として、SSNP は **非コンセンサス・外部レイヤー** のプログラムであり、
既存のハーベスト報酬、トランザクション手数料、ブロック報酬を変更しない。

## 1. 「結局これはハーベスト報酬を別名で削る話ではないか」

違う。SSNP の前提は、**既存ハーベスト報酬に手を付けない** ことである。

カウンター:

- SSNP の報酬原資は別枠で扱う。
- コンセンサス報酬と外部プログラム報酬は制度として別物である。
- 既存報酬を削る案は、別のガバナンス論点であり、SSNP ドラフトの対象外である。

要するに、これに反対するなら反対対象を取り違えている。ハーベスト改定案への反対と、外部インセンティブ案への反対は分けて議論すべきである。

## 2. 「Voting Node に報酬を付けると中央集権化する」

報酬設計が雑ならそうなる。だから SSNP は **性能条件** と **寡占防止条件** を同時に入れている。

カウンター:

- 単純な一律配布ではなく、Qualified 条件を満たしたノードだけを対象にする。
- Same Operator Group 単位で 1 枠に制限する。
- 同一ドメインの多重取得も制限する。
- 分散スコアを評価に入れ、集中を有利にしない。

中央集権化を理由に反対するなら、対案は「無制限複数枠取得を防ぐ具体ルール」を出すべきである。抽象論だけでは不十分である。

## 3. 「大規模運用者だけが有利で、小規模ノードは排除される」

半分正しいが、結論が雑である。雑な制度なら大規模運用者が勝つ。SSNP はその逆を防ぐために設計している。

カウンター:

- 1事業者が複数枠を総取りしにくい。
- 高額設備よりも、継続運用、監視応答、Voting Key 維持のような基本運用品質を重視する。
- 初期段階では通知と監視だけでも価値があるため、即時に資本力競争へ寄せる必要がない。

問題は「大規模運用者が参加すること」ではなく、「同一支配下の複数ノードが多重取りできること」である。SSNP はそこを叩いている。

## 4. 「監視スコアはゲーム化される。低遅延だけ偽装すれば勝てる」

その通り。だから単一指標で評価してはいけない。

カウンター:

- SSNP は availability、finalization tracking、chain sync consistency、Voting Key continuity の複合評価を前提にしている。
- 単純な ping 最速競争ではない。
- 外部監視を複数地域から行うことで、単一点最適化の価値を下げる。
- proxy や中継で見かけだけ整える行為は、監視設計と異常検知ルールの論点として明示的に扱うべきである。

「ゲーム化されうる」は反対理由ではなく、**測定設計を厳密にしろ** という設計要求である。

## 5. 「Same Operator Group の判定は恣意的で危険だ」

放置する方が危険である。実質同一運用主体を見ない制度は、必ず抜け穴になる。

カウンター:

- ドラフトは operator address、domain、managed provider、連絡先、証明書管理主体、運用権限など複数根拠を使う。
- 単一証拠で即断する前提ではない。
- 問題は判定概念の存在ではなく、証拠基準と異議申立て手続の整備不足である。

必要なのは「判定を捨てること」ではなく、「証拠レベル、レビュー手順、再審フローを定義すること」である。

## 6. 「同一ドメイン 1 枠は厳しすぎる。まともな運用まで巻き込む」

厳しめである。だが、公開ランキングに報酬を載せる以上、抜け穴は先に塞ぐ必要がある。

カウンター:

- ドメイン制限は完全な本人判定ではなく、安価な多重取得を防ぐ実務的フィルタである。
- 単独ではなく Same Operator Group 判定と組み合わせる前提で使う。
- 過剰なら将来緩和できるが、初期から緩いと制度信用が死ぬ。

初期制度は、取り返しのつきにくい「緩すぎる失敗」を避けるべきである。厳しすぎるなら運用実績を見て調整すればよい。

## 7. 「こんな外部ランキングに正統性はない。勝手に序列を作るな」

正統性の源泉を誤解している。SSNP は **プロトコル権限** を主張していない。外部基準を公開するだけである。

カウンター:

- コンセンサス参加資格を変えない。
- ノード排除権限を持たない。
- 外部プログラムとして、公開ルールに基づいて評価と支援を行うだけである。

市場やコミュニティがそのランキングを採用しなければ影響は限定的である。つまり、強制力ではなく説明責任で勝負すべき領域である。

## 8. 「通知機能まで入れると運営依存が強くなる」

通知は依存ではなく、事故率低下のための補助である。依存が問題なら、通知なし運用の方がむしろ脆い。

カウンター:

- 通知はコンセンサス制御ではない。
- 使わなくてもノード運用自体は可能である。
- 予防可能な障害を減らす副次機能として合理性がある。

反対するなら、「通知なしでも Voting Key 失効や証明書期限切れを十分防げる代替運用」を示すべきである。現実にはそこが弱い。

## 9. 「監視インフラ自体が集中したら本末転倒だ」

その懸念は正しい。これは SSNP の最大級の実装リスクの一つである。

カウンター:

- だからこそ監視インフラ分散は Known Open Questions に入っている。
- 監視実装はマルチリージョン前提にすべきである。
- 可能なら測定主体やデータ公開も分散させるべきである。

この論点の結論は「SSNP をやめろ」ではない。「監視の集中を許すな」である。反対ではなく、設計要件として取り込むべき指摘である。

## 10. 「今やるべきは制度ではなく、まずノード数を増やすことだ」

数だけ増やしても意味がない。低品質ノードや同一支配下ノードを増やしても、finalization 耐性は見かけほど改善しない。

カウンター:

- SSNP は単なる数ではなく、継続運用と独立性を重視する。
- 放置された低品質ノード群より、運用品質が見える Qualified Node 群の方が価値が高い。
- 通知、監視、ランキングは、単純なノード数拡大より実効性が高い。

「数を増やせ」は指標として粗すぎる。必要なのは、**落ちにくく、偏りにくい Voting Node を増やすこと** である。

## 結論

SSNP への反対論のうち、まともなものは大きく 2 種類しかない。

- 報酬原資をどうするか
- 寡占判定と監視設計をどこまで厳密にできるか

それ以外の多くは、制度の目的と境界を取り違えた反対である。

SSNP の論点は「やるかやらないか」ではない。正しく言えば、

- 既存報酬を触らず、
- 運用品質を可視化し、
- 寡占を抑え、
- 外部監視と通知で落下事故を減らす、

この設計をどこまで詰められるか、である。
