## 概要

本提案は、Symbolネットワークにおける **Voting Node の安定稼働、投票継続性、分散性** を向上させるための、性能ベースの監視・ランキング・インセンティブ枠組み **Symbol Super Node Program（SSNP）** を導入するものである。

SSNP は、コンセンサスルール、ハーベスト報酬、トランザクション手数料、ブロック報酬を変更しない。代わりに、**高品質な Voting Node 運用者を支援する独立した外部レイヤー** として機能する。プログラムは、外部から検証可能な運用指標に基づいてノードを評価し、寡占防止ルールを適用したうえで、上位の Qualified Node に対して別枠の資金源から報酬を配分する。

加えて、SSNP は、障害通知、証明書期限通知、Voting Key 更新通知などを運用者に提供し、予防可能な voting 離脱を減らす。

## 背景

Symbol の finalization は、有効な Voting Node の十分な参加に依存している。一方、現状には以下の構造的リスクがある。

1. Voting Node に、投票継続性に直接連動した報酬がない。
2. Voting Key は更新漏れにより失効しうる。
3. 運用が一部の主体や代行事業者に集中しうる。
4. 地理的・インフラ的分散が十分でない可能性がある。
5. 共通の公開・性能ベースの qualification 仕組みが存在しない。

したがって、**信頼性、継続性、分散性** を報いる制度は、プロトコル経済を変更せずにネットワーク耐性を高めうる。

## 目的

SSNP の目的は以下である。

1. 高品質な active Voting Node を増やす。
2. Finalization の耐性を向上させる。
3. Voting Key 期限切れや保守放置による静かな離脱を減らす。
4. 地理的・インフラ的分散を促進する。
5. 単一オペレーターや運用代行業者による報酬独占を防ぐ。
6. 公開ランキングと運用品質シグナルを作る。

## 非目的

SSNP は以下を目的としない。

- Symbol のコンセンサスルール変更
- オンチェーンスラッシング
- 全参加者への一律報酬
- 既存ハーベスト報酬の代替
- 人気投票や委任集中への報酬付け

## 定義

### Qualified Node

報酬期間において、以下の最低条件をすべて満たすノードを **Qualified Node** とする。

- 日次 availability 99.0%以上
- 有効 probe の 95%以上で finalized lag が閾値内
- 有効 probe の 95%以上で chain sync が閾値内
- 対象 current epoch に対して有効な Voting Key を保持
- Program Agent の heartbeat が有効
- 重大な異常や確認済み不正がない

### Same Operator Group（同一オペレーターグループ）

以下を含む、実質的に重なった運用支配下にあると合理的に判断されるノード群を **Same Operator Group** とする。

- 同一 operator address
- 同一 registrable domain
- 同一マネージドホスティングまたは運用代行業者
- 同一連絡先または証明書管理主体
- 同一の運用権限者または委託運用チーム

報酬対象選定において、Same Operator Group 内のノード群は 1 グループとして扱う。

## システム概要

SSNP は以下で構成される。

- 公開ポータル
- 登録と challenge 署名フロー
- マルチリージョン外部監視
- heartbeat 保証と補助テレメトリのための Program Agent
- スコアリング・ランキングエンジン
- 通知システム

外部監視を主たる真実源とし、Program Agent のデータは補助情報に留め、外部計測と矛盾する場合に優先させてはならない。

## 登録

参加オペレーターは最低でも以下を満たす。

- Symbol mainnet の node operator であること
- current epoch に有効な Voting Key を保持していること
- 監視対象 endpoint を公開していること
- Program Agent を実行していること
- 利用規約に同意していること

そのうえで、オペレーターは以下を提出する。

- operator address
- ノード識別情報
- Voting Key の検証に必要な情報
- 監視対象 endpoint 情報
- 通知用連絡先
- 支配証明のための署名付き challenge
- Program Agent の連携情報

新規登録ノードは、Qualified となる前に最低 72 時間の観測期間に入る。

ここで重要なのは、単独のハードウェア性能ではなく、そのノードが現在進行形で安全かつ可観測な voting を継続できるかである。

## スコアモデル

総合スコアは以下とする。

`S = 0.7 * B + 0.3 * D`

- `B` = ベース性能スコア
- `D` = 分散スコア

### ベース性能スコア（70%）

- Availability: 30
- Finalization追従: 20
- Chain sync 一貫性: 10
- Voting Key 継続性: 10

### 分散スコア（30%）

- 地理的分散: 15
- ASN / インフラ分散: 10
- 国集中回避: 5

分散スコアは、Qualified である性能を代替するものではなく、性能を満たしたノード同士の補正である。

## ランキング

Qualified Node を総合スコアの降順で並べる。

報酬対象候補は最大 50 ノードとし、さらに寡占防止フィルタを適用する。

同点の場合の優先順は以下とする。

1. Finalization スコアが高い方
2. Availability スコアが高い方
3. 検証済み登録時刻が早い方

## 寡占防止ルール

### Same Operator Group あたり 1 枠

同一報酬期間において、Same Operator Group から報酬対象に選ばれるノードは最大 1 台までとする。

### 同一 registrable domain あたり 1 枠

同一 registrable domain を使用するノードは、同一報酬期間に最大 1 台までを報酬対象とする。

### 運用代行事業者

同一のマネージドホスティング事業者、ノード運用代行事業者、または委託運用業者により実質的に運用支配が共有されるノード群は、Same Operator Group の一部として扱う。

同一グループから複数ノードが上位に入った場合、最上位ノードのみを報酬対象とし、残りは次順位の独立ノードで繰り下げ補充する。

## 報酬原資

SSNP は **独立した資金源** を用い、既存のハーベスト報酬、手数料、ブロック報酬を削減しない。

`R_daily` を名目上の日次報酬総額とする。

### Participation Adjustment

対象期間の Qualified Node 数を `N` とする。

実際に放出する割合は以下とする。

- N = 1–9  -> 30%
- N = 10–19 -> 50%
- N = 20–29 -> 70%
- N = 30–39 -> 85%
- N = 40–49 -> 95%
- N >= 50 -> 100%

したがって、

`R_dist = R_daily * P(N)`

未放出分は将来期間または承認済み用途のために留保する。

これは制度初期の過大配分を防ぎ、Qualified Node 数を増やす方向にインセンティブを合わせるためである。

## 順位帯ごとの配分

実放出額 `R_dist` は以下の順位帯で配分する。

- 1–5位: 25%
- 6–10位: 20%
- 11–20位: 25%
- 21–35位: 18%
- 36–50位: 12%

各順位帯の中では均等分割する。

## 通知機能

SSNP は登録済み運用者に対し、少なくとも以下の運用通知を提供することが望ましい。

- ノード障害通知
- Finalized lag 通知
- Voting Key 期限通知
- TLS 証明書期限通知
- ドメイン期限通知
- ソフトウェア更新推奨通知
- Program Agent heartbeat 異常通知

通知機能の目的は、回避可能な qualification 脱落を減らし、ネットワーク継続性を高めることにある。

## 資金方針

初期導入は、外部資金または別途承認されたプログラム予算で行う。ハーベスト報酬の削減は本ドラフトの対象外とする。

## 導入計画

### Phase 1
監視、登録、公開ランキング、通知のみ。

### Phase 2
別予算による手動または半手動の報酬配布。

### Phase 3
ガバナンス承認を前提とした自動計算・自動配布。

## セキュリティ考慮

- 外部計測を主とする
- 登録時に署名を必須とする
- proxy や mirror endpoint の悪用を監視する
- Program Agent の自己申告値を外部計測より優先させない
- 集中判定はドメイン単独でなく複数証拠を使う

## 提案理由

本プログラムは、単なる参加ではなく、**ネットワークを支える行動** に報いるよう意図されている。すなわち、測定可能な uptime、Voting 継続性、finalization 支援、地域・インフラ分散を重視する。

また、委任人気への報酬付けは集中を強める可能性が高いため、意図的に採用しない。

## 後方互換性

本提案はプロトコル変更を必要とせず、既存の Symbol ノード運用と後方互換性を持つ。

## 結論

SSNP は、Symbol の運用耐性を高めるための、現実的で、非コンセンサス型で、分散配慮型の枠組みである。Voting Node に関する実際のリスクに対応しつつ、既存の harvesting incentive と正面衝突しないよう設計されている。
