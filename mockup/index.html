Program Snapshot

SSNP が扱う対象

このモックアップは仕様書のMVP前提に限定する。対象は Symbol mainnet の voting node であり、自己申告ではなく外部 probe を主な真実源として扱う。

登録状態

新規ノードは登録後に 72 時間の観測期間へ入り、Qualified 判定前に継続性を確認する。

  • 登録済みだが未観測
  • 観測済みだが未 Qualified
  • Qualified だが報酬対象外

Qualification Gate

性能ランキングは Qualified 後に使う。合格条件の代替にしてはいけない。

  • Availability 99.0%以上
  • Finalized lag 閾値内 95%以上
  • Chain sync 閾値内 95%以上
  • Voting key 有効、Agent heartbeat 正常

Reward Guardrail

寡占防止を先に掛ける。Same Operator Group と同一 domain の多重取得を防ぐ。

  • Group ごとに最大 1 枠
  • Registrable domain ごとに最大 1 枠
  • 残りは独立ノードで backfill

Registration

登録フローのモック

実装対象は登録ポータルの入力設計であり、署名検証や API 実装はまだ持たせない。 まず入力項目と状態遷移を固定する。

入力必須項目

状態遷移

  1. Registered

    利用規約同意、endpoint 提出、署名 challenge 提出。

  2. Observed (0-72h)

    マルチリージョン probe が availability / sync / finalized lag を計測。

  3. Qualified

    current epoch の voting key、heartbeat、重大異常なしを確認。

  4. Reward Eligible or Filtered

    寡占防止ルールで reward eligibility を別判定。

運用原則

  • 外部観測結果を優先し、自己申告で上書きしない
  • 1 リージョン障害で全評価を無効化しない
  • Qualified 判定と reward eligibility を分離する
  • 通知配送失敗も運用リスクとして監視する

Architecture

低コスト運用を前提にした実装方針

モックアップ段階ではフロントエンドを静的ファイルに限定する。将来は JSON や API に差し替え可能だが、 現時点では表示層だけを明確に分離する。

今すぐ持つもの

  • 静的 HTML
  • 単一 CSS
  • 依存ゼロの素の JavaScript
  • ダミーデータ駆動のテーブル描画

後で差し替えるもの

  • ランキング JSON/API
  • 登録送信 API
  • 認証・署名検証
  • 通知イベント配信

避けるもの

  • フレームワーク依存の初期固定化
  • ビルド必須構成
  • モック段階での過剰なバックエンド実装
  • CPU ベンチや ping 中心の誤指標
